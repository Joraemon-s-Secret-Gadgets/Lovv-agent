# V2_32 Alternative Itinerary Weather Directive

> Status: design directive, implementation pending.
> Scope: use `src/lovv_agent_v2/resources/city_monthly_weather_risks.json` to decide when and how V2 exposes a weather-aware alternative itinerary.

## 1. Decision

`alternative_itinerary` is not a second full itinerary generated by default.
It is a weather-risk backup surface that is offered only when the selected city and travel month make the primary itinerary meaningfully weather-sensitive.

Primary itinerary generation remains unchanged:

```text
city_select / direct anchor
-> planner retrieve_places
-> route_days
-> assemble_itinerary
-> response_packager
```

Weather risk is evaluated after the primary itinerary is assembled, because the trigger depends on both:

- selected city and `travel_month`
- actual itinerary outdoor/mixed exposure

## 2. Weather Risk Resource Contract

Runtime source:

```text
src/lovv_agent_v2/resources/city_monthly_weather_risks.json
```

Lookup key:

```text
(city_id, travel_month)
```

Important fields:

```json
{
  "city_id": "KR-...",
  "month": 7,
  "risk": {
    "overall": "low | medium | high",
    "dimensions": {
      "rain": "low | medium | high",
      "heat": "low | medium | high",
      "cold": "low | medium | high",
      "low_sunshine": "low | medium | high",
      "snow": "low | medium | high | unknown"
    },
    "scores": {},
    "reason_codes": []
  },
  "data_quality": {}
}
```

Coverage caveat:

- The table currently covers 68 climate-table cities and 816 city-month rows.
- Missing city-month risk is normal and must not fail planner.
- Missing risk means `weather_risk_status = unavailable`, no alternative itinerary prompt.

## 3. Trigger Policy

Weather notice and alternative generation are separate.

| Condition | Planner behavior |
|---|---|
| no risk row | no weather notice, no alternative itinerary |
| `overall=low` | no alternative itinerary |
| `overall=medium` and outdoor/mixed ratio >= 40% | add weak weather notice |
| `overall=high` and outdoor/mixed ratio >= 25% | add strong weather notice |
| `overall=high` and outdoor/mixed ratio >= 50% | offer weather alternative |

The exact percentages follow `V2_16` and may be tuned after smoke measurement.

User-facing wording must say "평년 기준 리스크" and must not imply a live forecast.

## 4. Outdoor Exposure Classification

Planner should classify each itinerary item into:

```text
indoor | outdoor | mixed | unknown
```

Initial classifier should be deterministic and auditable:

1. `item_type == festival` -> `outdoor` unless payload later carries indoor evidence.
2. `attraction_subtype_code` based classification from `attraction_subtypes.json`.
3. title/theme fallback only when subtype is absent.
4. unknown stays unknown and should not count as outdoor exposure.

Risk-sensitive dimensions:

| dimension | affects |
|---|---|
| `rain` | outdoor/mixed slots |
| `heat` | outdoor/mixed daytime slots |
| `cold` | outdoor/mixed slots |
| `snow` | outdoor/mixed slots, especially walking routes |
| `low_sunshine` | outdoor scenic/view slots, weaker signal |

## 5. Alternative Itinerary Semantics

`alternative_itinerary` is a replacement proposal for weather-sensitive slots, not a city replacement.

Rules:

- Keep selected city.
- Keep trip type and day count.
- Keep seed/festival items unless the seed itself is unsafe and user explicitly asks to replace it.
- Replace only affected outdoor/mixed non-seed slots.
- Prefer indoor or weather-resilient subtypes.
- Preserve day/order shape where possible.
- Do not silently replace primary itinerary.

Recommended alternative item fields reuse the normal planner item shape plus weather metadata:

```json
{
  "day": 1,
  "slot": "afternoon",
  "item_type": "attraction",
  "placeId": "...",
  "title": "...",
  "body": "...",
  "reason": "비가 많은 달이라 실내 후보로 대체할 수 있습니다.",
  "replacesItemId": "item-2",
  "weatherRiskReason": "rain_high",
  "weatherRiskDimensions": ["rain"]
}
```

## 6. Response Contract

Public response remains the recommendation response envelope.

Primary path:

```text
response_status = modification_pending
itinerary.days = primary itinerary
alternativeItinerary = optional weather backup proposal
explainability.userNotice includes weather notice
```

If implementation wants explicit user choice before generating backup items, use existing clarification mechanics:

```text
reason_code = weather_alternative_available
then = weather_alternative
```

However, V2 first implementation should prefer a lightweight precomputed `alternative_itinerary` when enough reserve/indoor candidates are already available. If no credible replacements exist, return only the notice.

## 7. Planner Integration Point

Add the weather step after `assemble_itinerary` has built the primary itinerary and before response packaging/explanation:

```text
route_days
-> assemble_itinerary
-> weather_alternative
-> explain_itinerary
-> response_packager
```

Reason:

- It needs final slots and route item payloads.
- It should not perturb selection/routing of the primary itinerary.
- It can reuse planner reserve and selected city/month metadata.

State additions:

```text
state.planner.weather_risk
state.planner.weather_notice
state.planner.alternative_itinerary
state.planner.validation_result.weather_audit
```

## 8. Candidate Source For Alternative Slots

Use candidate sources in this order:

1. route/selection reserve from same city
2. planner place pool candidates not selected
3. same-city S3 Vector re-query with indoor/weather-resilient subtype hints

Do not use automatic alternative-city fallback for weather alternatives.

The first implementation can start with reserve/place-pool only. S3 re-query can follow once replacement quality is measured.

## 9. Initial Implementation Plan

1. Add weather risk resource loader keyed by `(city_id, month)`.
2. Add deterministic item exposure classifier.
3. Add `weather_alternative` planner step that produces notice + optional alternatives.
4. Add packager support for `alternativeItinerary` if not already exposed.
5. Add smoke cases:
   - high-rain month + outdoor-heavy itinerary -> notice + backup
   - low-risk month -> no notice
   - missing risk city -> no failure
   - festival seed present -> seed preserved

## 10. Non-goals

- No live weather forecast.
- No automatic city replacement.
- No automatic primary itinerary mutation.
- No exposure of raw percentile `reason_codes` to users.
- No broad reranking of city_select by weather risk in V2 first pass.
