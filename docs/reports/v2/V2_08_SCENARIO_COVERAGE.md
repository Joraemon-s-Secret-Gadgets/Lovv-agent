## 0. 한 문장 요약
> V2는 시나리오를 **세 흐름**으로 처리한다 — 다이어그램의 세 영역과 1:1로 맞는다.
> **① 초회 생성(상단 Main Graph)** · **② 자연어 수정 루프(하단 우측)** · **③ 데이터 계약(하단 좌측)**.
> 시나리오는 곧 (a) 처음 만들기, (b) 고치기, (c) 막다른 상황 처리 셋으로 나뉘고, 아래에서 그 순서대로 본다.

세 흐름의 처리 상태를 미리 깔면: **[V1·고도화]** 8건은 "되던 게 좋아짐"(거의 V2.0), **[V2·신규]** 9건은 "새로 됨"(절반은 V2.0 thin slice, 절반은 V2.1 백로그).

## 1. 초회 생성 경로 — 다이어그램 상단을 왼→오로 따라가며

### Intent (요청 해석 · 모순→되묻기)
입력의 형태 문제는 여기서 다 걸러진다.
- **모호 입력**("나한테 맞는 데로", SC-01): 프로필이 충분하면 채우고, 아니면 되묻기 — 분기 자체가 다음 노드(Profile)에서 결정된다.
- **모순 입력**("조용한데 사람 많은 데", SC-N2): **무조건 되묻기**(`needs_clarification`). 창의적 절충은 안 한다 — 엉뚱한 결과를 막기 위한 확정 정책.
- **긴 서사**("회사도 힘들고 외롭고 바다는 보고싶고"): 핵심 키워드만 뽑아 진행.
- **이동수단**("뚜벅이로", SC-N1): `transport_pref = walk/car/unknown` 거친 신호로 변환 → 배치 단계에서 작동.

### Supervisor (초회·resume 분기 · 수정 의도 분해)
허브 라우터. 초회면 city_select로 보내고, resume면 수정 경로로 분기한다. 수정일 때 **여러 편집을 op 리스트로 분해**하는 것도 여기 책임(§2에서 다시 등장).

### city_select subgraph (2-node) — V2 품질 개선의 핵심
`retrieval_node`(S3 Vector 검색·병합) → `scoring_and_selection_node`(점수·도시 확정·seed 추출).
- **멀티테마 발견**(SC-G1): 테마 게이트를 **AND→soft**로 바꾼 게 V1 최대 약점의 해소. "산이랑 바다 같이"에서 소도시가 탈락하지 않고 살아남는다. **이 한 줄이 V2 가치의 절반.**
- **감정·저밀도**(번아웃 SC-00, 이별, 한적): congestion(혼잡도) 가중으로 한산한 곳을 띄운다.
- **개인화**(SC-01 A안): Profile의 `theme_weights`를 점수에 실제로 반영.
- **도시 지정**(SC-G2): anchored 모드면 검색 풀을 그 도시로 고정.
- **seed 추출**: 도시를 고른 이유가 된 대표 장소 = 각 day의 anchor. 이후 배치·수정이 전부 이 seed를 축으로 돈다.
- capacity(candidate_sufficiency) 결합은 **제거** — 점수는 도시 적합도만 보고, 장소 부족은 Planner가 흡수한다.

### Festival Verifier (optional)
축제 포함 요청(SC-G3)일 때만 켜진다. 축제 날짜를 검증해 **실제 날짜에 분산 배치**(V1은 전부 day-1 오후 고정이었음). 시즌 불일치(SC-R2)면 축제만 빼고 안내. 테마 정합 필터는 축제 테마 태깅 데이터가 적재된 경우 작동.

### Planner (초회 2-pass + on-demand Plan B)
Pass1(도시·테마 게이트) → Pass2(도시 고정·테마 off로 PlacePool 확보) → **seed 라운드로빈 배치 + 거리 페널티(haversine)**.
- **후보 부족**(SC-R1): Pass2 재인출로 먼저 채워 축소 빈도를 낮춘다.
- **동선**(SC-00): seed anchor + geo로 순서를 잡는다. 장소 간 이동시간 표시는 front 담당.
- **이동수단**(SC-N1): walk면 거리 페널티 강화(도보 집약), car면 완화.
- **날씨**(SC-02): 기상 임계 초과 시 일정은 그대로 주되 `weatherNotice`로 "비 잦아요, 실내로 바꿔드릴까요?"를 띄운다. 실내 대안은 **요청하면 그때** 만든다(on-demand) — 미리 두 벌 만들지 않는다.

### Response Packager
결과를 5종 상태로 포장: 정상/축소(`completed`) · no_candidate 재질의·안전거부(`END_WAIT_USER`, SC-R3/R4). 그리고 interrupt를 걸어 checkpoint 저장 → 사용자의 다음 말(수정)을 기다린다. 여기서 **수정 루프로 넘어간다.**

## 2. 자연어 수정 루프 — 다이어그램 하단 우측

resume(thread_id) → 수정 Intent(**edit_ops 분해**) → 교체 전용(seed 제외) → 재인출 → **일괄 재배치**.

- **슬롯 1개 교체**(SC-03, 1차 핵심): "2일차 오후 바다 보이는 조용한 데로" → 그 슬롯만 S3 재인출(조건=무드·타입·위치를 추가 query로), seed·배치개수(≤3)는 고정.
- **다건 동시 수정**(신규): "2일차는 카페로, 3일차는 바다로" → op 리스트로 분해 후 각자 재인출 → **한 번에 적용하고 단일 재배치**(순차 적용 안 함 = 동선 꼬임 방지). 일부 슬롯만 후보가 없으면 **성공분은 적용 + 실패분은 안내**(`modification_pending`).
- **모순된 다건**: op 간 충돌이면 멈추고 되묻기 — §1 Intent의 모순 정책과 동일하게 동작.
- **전면 리셋**(SC-M4): "다 별로야" → 기존 도시·테마를 **세션 avoid**로 두고 차순위 재생성. 이 기피는 그 세션이 끝날 때까지 유지되고 영구 프로필엔 안 남는다.
- **날씨→실내**(SC-02 후속): weatherNotice에 사용자가 동의하면 이 루프를 타고 실내 대안(`alternativeItinerary`)을 생성.
- **도시 변경**(SC-M1): 수정 중 **유일하게 city_select까지 거슬러 올라가는** 경로. V2.0은 경로만 설계하고 구현은 V2.1.

상태값은 V1의 `completed`/`END_WAIT_USER`에 **`modification_pending`** 하나만 더했다(수정 완료 + 다음 수정 대기).

## 3. 막다른 상황은 어떻게 막나 (엣지 — V1 5종 유지)
불가능 동선("강원·경북 다 당일치기") → 축소 안내 · 범위 밖("제주/강남역") → 강원·경북 한정 재질의 · 이상값("영하 100도") → 안전 정정 · 축제 시즌 불일치 → 고지+대안 · 후보 0매칭 → 재질의 · 필수필드 누락 → 안전 거부. 전부 graceful 안내 + 상태값으로 끝낸다.

**범위 밖 요청**(숙소·식당·예약·실시간 혼잡·내비·사진/음성 입력)도 들어올 수 있다 → Intent가 인식해 **"그건 못 한다"고 안내**하는 데까지가 V2 책임(기능 구현은 안 함). 조회·지도 동선 시각화는 front, 안전 정책은 별도 트랙.

## 5. 그래서 V2.0에서 실제로 보이는 것
**thin slice = 초회 품질 개선(soft 테마·seed 배치·capacity 제거) + 슬롯 교체 수정(단일·다건) + 날씨 안내.**
시연 한 줄로: *"멀티테마로 소도시를 제대로 찾아주고 → 만든 일정을 자연어로 '여기 바꿔줘'(여러 개도) 하면 고쳐주고 → 비 오는 달이면 안내까지 한다."*

**다음 사이클(V2.1)**: 날씨 실내 대안 **실제 생성**, 도시 변경, 무드/길이/날짜 수정, 프로필 학습 체감, 이동수단 강도 튜닝.

## 6. 미팅에서 "이건요?" 들어올 만한 것 — 미리 답
- *"여러 개 한 번에 고치면?"* → op 분해 + 일괄 재배치, 일부 실패는 부분 적용+안내(§2). **모순이면 되묻기.**
- *"비 오는 날 두 벌 다 만들어?"* → 아니다. 안내만, 원하면 그때 생성(on-demand).
- *"프로필은 언제 학습?"* → 일정을 **저장**할 때만. 수정 중 변덕은 안 쌓는다.
- *"숙소/예약/지도는?"* → 범위 밖. 숙소·예약은 "안내만", 지도 동선은 front.
- *"4박5일은?"* → 짧은 일정 품질 먼저 확인 후 확장(요청은 받되 한계 고지).
